# CloudFront & S3

## S3 정적 웹 호스팅

- S3는 AWS의 스토리지 서비스로, **정적 웹 호스팅** 기능을 통해 웹 서버처럼 사용 가능
- 정적 파일(HTML, CSS, JS, 이미지 등)을 URL로 접근할 수 있게 해주는 것
- 단순히 파일을 서빙하는 것이기 때문에 **경로와 파일이 1:1 매칭**

### S3 단독의 한계

- 버킷이 생성된 **오리진 리전에서만 서빙** → 먼 지역 사용자일수록 레이턴시가 길어짐
- 요청마다 비용 발생
- 커스텀 도메인에서 **HTTPS 적용 불가**
- 퍼블릭으로 열려있어야 해서 보안 이슈 가능성

---

## SPA와 S3의 문제

SPA는 `index.html` 하나를 진입점으로 사용하고, JS가 URL을 읽어서 화면을 동적으로 그려냄.
즉, `/about`, `/profile` 같은 경로는 **실제 파일이 아니라 JS가 만들어낸 가상의 경로**.

```
/index.html → ✅ 파일 있음
/about      → ❌ 파일 없음 → 404
/profile    → ❌ 파일 없음 → 404
```

경로와 파일이 1:1 매칭이 아니기 때문에 S3 단독으로는 SPA 라우팅 처리가 어려움.

---

## CloudFront

AWS CDN(Content Delivery Network) 서비스. 전 세계 450+ 엣지 로케이션에 콘텐츠를 캐싱해서 사용자와 가장 가까운 엣지에서 응답.

---

## CloudFront + S3 연동

S3를 **오리진(원본 서버)** 으로 두고, CloudFront가 앞단에서 요청을 받아 서빙.

```
사용자 → CloudFront(엣지) → S3(오리진)
```

- S3 버킷은 **비공개**로 유지, CloudFront(OAC)만 접근 허용
- 엣지마다 캐싱 처리 → 캐시 히트 시 S3까지 요청 안 올라감 → 비용 절감 + 속도 향상
- CloudFront + ACM으로 **커스텀 도메인 HTTPS 적용** 가능

### SPA 라우팅 해결

Custom Error Response 설정으로 완전히 해결

```
S3 단독 → 404 발생 시 index.html로 리다이렉트하지만 상태코드가 404 그대로
CloudFront → 404 → 200 + index.html 반환 (완전한 해결)
```

### 캐시 전략

```
index.html        → Cache-Control: no-cache (항상 최신 버전)
*.js, *.css (해시) → Cache-Control: max-age=31536000, immutable (1년)
```

### Invalidation (무효화)

캐시된 데이터를 날리고 새로 캐싱하는 것. 배포 후 즉시 반영이 필요할 때 사용.
단, 비용이 발생하기 때문에 실무에서는 파일명에 **content hash**를 붙여 캐시 버스팅하는 방식을 선호.

### 압축

1KB ~ 10MB 사이의 압축 가능한 텍스트 파일(HTML, CSS, JS 등)을 클라이언트가 `Accept-Encoding` 헤더를 보낼 때 gzip/br로 압축해서 서빙.
