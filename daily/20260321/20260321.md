# Babel → SWC 전환으로 번들 크기가 5배 줄어든 이유

> 실제 프로젝트에서 경험한 내용을 면접에서 설명하기 위한 정리입니다.

---

## 한 줄 요약

> Babel은 코드를 변환할 때 "불필요한 짐"을 많이 싣고 다녔는데, SWC는 그 짐을 대부분 없애줬다.

---

## 비유로 이해하기

- **Babel** = 오래된 번역가. 모든 문장을 가장 오래된 독자도 이해할 수 있게 풀어쓴다. 그 과정에서 원문보다 훨씬 긴 글이 된다.
- **SWC** = 최신 번역가. 최신 독자(모던 브라우저)를 타겟으로 하면, 굳이 풀어쓰지 않아도 되는 부분은 그냥 둔다.

---

## 구체적으로 무슨 일이 일어났나?

### 1. Babel 헬퍼 함수가 사라졌다

Babel은 최신 자바스크립트 문법을 오래된 브라우저용으로 변환할 때, 자체 "도우미 코드(헬퍼)"를 코드 곳곳에 주입한다.

```js
// Babel이 class 문법을 변환한 결과 (실제로 이런 코드가 번들에 들어감)
var _classCallCheck = require("@babel/runtime/helpers/classCallCheck");
var _createClass = require("@babel/runtime/helpers/createClass");
```

프로젝트 규모가 클수록 이 헬퍼 코드들이 수십~수백 개 쌓인다.
SWC는 이런 헬퍼를 거의 생성하지 않는다.

---

### 2. styled-components 처리 방식이 달라졌다

이 프로젝트는 `styled-components`를 많이 사용한다.

- **Babel**: `babel-plugin-styled-components`가 런타임에 동작하면서 추가 코드를 번들에 포함시켰다.
- **SWC**: `next.config.js`의 `compiler.styledComponents` 옵션으로 **컴파일 타임에 처리**하여 런타임 코드가 필요 없어진다.

```js
// next.config.js - SWC가 styled-components를 직접 처리
compiler: {
  styledComponents: {
    displayName: true,
    ssr: false,
  },
},
```

---

### 3. @syncfusion 같은 대형 라이브러리의 tree-shaking이 잘 됐다

이 프로젝트에는 `@syncfusion` 패키지가 12개 이상 있다. 매우 큰 라이브러리다.

- **Tree-shaking**: "실제로 쓰는 코드만 번들에 포함시키는 것"
- Babel은 ES Module 구조를 변환하면서 tree-shaking을 방해하는 코드를 만들기도 했다.
- SWC는 ES Module 구조를 그대로 보존해서 번들러가 **불필요한 코드를 더 잘 제거**할 수 있다.

## 면접에서 이렇게 말하기

> "Babel에서 SWC로 전환하면서 app.js 번들 크기가 약 5배 줄었습니다.
>
> 주요 원인은 세 가지였는데요.
>
> 첫째, Babel이 코드 변환 시 자동으로 주입하던 `@babel/runtime` 헬퍼 코드들이 제거됐습니다.
>
> 둘째, 프로젝트에서 `styled-components`를 많이 쓰는데, 기존엔 Babel 플러그인이 런타임 코드를 추가했지만 SWC는 컴파일 타임에 처리하여 런타임 오버헤드가 없어졌습니다.
>
> 셋째, SWC는 ES Module 구조를 잘 보존해서 `@syncfusion` 같은 대형 라이브러리에 대한 tree-shaking 효율이 높아졌습니다."

## 정리표

| 원인              | Babel               | SWC              | 효과             |
| ----------------- | ------------------- | ---------------- | ---------------- |
| 런타임 헬퍼 코드  | 수십~수백 개 주입   | 거의 없음        | 번들 크기 감소   |
| styled-components | 런타임 처리         | 컴파일 타임 처리 | 런타임 코드 제거 |
| Tree-shaking      | ES Module 구조 훼손 | ES Module 보존   | 불필요 코드 제거 |
| 다운레벨 변환     | 모든 문법을 ES5로   | 필요한 것만 변환 | 변환 코드 감소   |

## Next.js의 app.js 파일이란?

### 정확한 이름은 `_app.tsx`

`pages/_app.tsx`는 **모든 페이지의 공통 래퍼** 역할을 하는 파일이다.

```tsx
// pages/_app.tsx
export default function App({ Component, pageProps }) {
  return <Component {...pageProps} />
}
```

모든 페이지가 이 컴포넌트를 통해 렌더링된다.

### 주로 하는 역할

| 역할 | 예시 |
|------|------|
| 전역 CSS 적용 | `import '../styles/global.css'` |
| 전역 레이아웃 | 공통 Header, Footer 감싸기 |
| 전역 상태 주입 | Recoil, Redux Provider |
| 인증 처리 | 로그인 상태 체크 |
| i18n 설정 | next-i18next appWithTranslation |

### 빌드 결과물에서 말하는 `app.js`는 다르다

`next build` 후 생성되는 `app-[hash].js`는 **`_app.tsx`에서 import한 모든 공통 코드의 번들**이다.

```
.next/static/chunks/
├── app-abc123.js   ← 공통 코드 번들 (이게 "app.js")
├── pages/index.js  ← 각 페이지별 코드
└── framework.js    ← React, ReactDOM
```

`_app.tsx`에서 import하는 게 많을수록 `app.js` 크기가 커진다.

### 이 프로젝트에서 app.js가 특히 많이 줄어든 이유

`_app.tsx`에서 `styled-components Provider`, `Recoil`, `next-i18next` 등이 모두 import되어 있고, 그것들의 런타임 코드가 전부 `app.js`에 들어간다.

- `app.js`는 **모든 페이지에서 공통으로 로드**되는 파일이다.
- 따라서 Babel → SWC 전환 시 공통 코드의 헬퍼와 런타임 오버헤드가 한꺼번에 제거되면서 **가장 드라마틱하게 줄어든 것**이다.
- 성능 영향도 가장 크다. 모든 페이지 첫 로딩에 영향을 주기 때문이다.
