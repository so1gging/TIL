# 2026.03.07

## Axios 취소 로직과 HTTP/1.1 한계

지도 이동 시 이전에 호출한 API를 취소하기 위해 AbortController(axios 취소) 기능을 넣었음.  
그런데 지도를 많이 움직이면 네이버 지도 타일이 보이지 않으면서 버벅이는 현상이 발생했고, 원인은 **HTTP/1.1**이었음.

### 왜 HTTP/1.1에서 문제가 되는가

- **Abort 시 연결이 끊김 → 매번 TCP 핸드셰이크부터 다시**: 요청을 abort하면 해당 TCP 연결이 끊어진다. 다음 요청은 **새 연결**을 맺어야 하므로 TCP 3-way handshake(그리고 HTTPS면 TLS 핸드셰이크)를 처음부터 다시 해야 한다. 지도를 자주 움직이면 “abort → 새 요청 → 또 abort”가 반복되면서 **매번 핸드셰이크 비용(RTT 수 배)**이 붙어, 체감상 버벅임이 커진다.
- **동시 연결 수 제한**: 브라우저는 **같은 도메인(origin)당 HTTP/1.1 연결을 보통 6개**만 허용한다. 지도 이동 시 타일 요청 + 우리 API 요청이 같은 도메인을 쓰면, 이 6개 연결을 모두 나눠 써야 한다.
- **취소 요청이 연결을 점유**: 요청을 abort해도 연결이 즉시 정리되지 않고, 취소 처리·연결 종료·재사용 가능 상태로 돌아오기까지 시간이 걸린다. 지도를 빠르게 움직이면 “취소 → 새 요청 → 또 취소”가 반복되면서 **연결 풀이 취소/재연결로 바쁘게** 돌아간다.
- **타일 요청이 밀림**: 실제로 화면에 필요한 건 **지도 타일 이미지 요청**인데, 6개 한도 안에서 API 요청과 경쟁하게 되고, 취소 로직 때문에 요청이 더 자주 바뀌면서 타일 요청이 큐에 밀리거나 지연된다. 그래서 타일이 안 보이거나 버벅이는 현상이 발생한다.
- **HTTP/2였다면**: HTTP/2는 한 연결 위에 여러 요청을 멀티플렉싱하므로, 연결 수 제한의 영향이 작고 요청 취소도 같은 연결 안에서 처리되어, 이런 현상이 덜했을 가능성이 크다.

정리하면, **같은 도메인에 대한 연결 수가 제한된 HTTP/1.1 환경에서** abort를 과하게 쓰면 오히려 연결 풀만 혼잡해져서, 지도 타일 같은 중요한 요청이 밀리면서 성능이 나빠진다.
