# 채용 과제 핵심 교훈

> NDA 준수: 회사명 및 서비스 도메인은 변경했습니다.

---

## 1. 채용 과제는 실무 프로젝트가 아니다

심사관은 코드를 30분 안에 읽는다.
파일 수가 많을수록 흐름 파악이 어려워지고, "왜 이렇게 복잡하게 짰지?"라는 의문을 낳는다.

실무에서라면 `api/catalog/api.ts`, `api/catalog/query.ts`, `api/catalog/model.ts` 3단 분리는 좋은 설계다.
하지만 3개 페이지짜리 과제에서 이 구조는 불필요한 파일 증가로 평가받을 수 있다.

**기억할 것:**
코드의 목적지가 "프로덕션 유지보수"가 아니라 "30분 심사"라면, 파악 속도가 우선이다.
좋은 구조란 항상 절대적인 게 아니라 **맥락에 맞는 구조**다.

---

## 2. "알고 있다"를 증명하려다 "판단력 없다"가 된다

React Query, Jotai, 컴파운드 컴포넌트, render props, 커스텀 훅 분리...
이 기술들을 안다는 건 좋은 일이다. 그런데 과제 규모에서 전부 꺼내 쓰면 역효과다.

심사관이 보는 건 두 가지다:
- 요구사항을 정확히 구현했는가?
- 이 규모에 맞는 판단을 했는가?

Zustand 하나로 충분한 장바구니 상태 관리를 React Query + Jotai 조합으로 구현하면,
"기술은 많이 아는데, 언제 쓸지 모르는 사람"으로 읽힌다.

**기억할 것:**
기술 스택 선택은 "무엇을 아는가"가 아니라 "왜 이걸 골랐는가"로 평가된다.
선택에 맥락이 없으면 오버엔지니어링이 된다.

---

## 3. 엣지케이스 처리가 진짜 실력을 드러낸다

구현의 완성도는 "정상 흐름"이 아니라 **"경계 조건"** 에서 갈린다.

장바구니 중복 처리 예시:
- 옵션 A → 옵션 B 순서로 선택
- 옵션 B → 옵션 A 순서로 선택

단순 비교(`JSON.stringify(options)`)로는 이 두 개를 다른 아이템으로 인식한다.
`options`를 `optionId` 기준으로 정렬한 뒤 직렬화해야 동일한 키가 나온다.

```typescript
// 잘못된 방식 - 선택 순서가 다르면 다른 키로 인식
const key = `${itemId}-${JSON.stringify(options)}`;

// 올바른 방식 - 정렬 후 직렬화
const sortedOptions = [...options].sort((a, b) => a.optionId - b.optionId);
const key = `${itemId}-${JSON.stringify(sortedOptions)}`;
```

이런 한 줄 차이가 실무 감각의 유무를 보여준다.

**기억할 것:**
과제를 구현한 뒤 "이 기능에서 사용자가 예상치 못한 행동을 하면?"을 한 번 더 물어보자.
엣지케이스를 건드리는 코드 한 줄이 수십 줄의 추상화보다 인상적이다.

## 4. 데이터는 "언제 계산할 것인가"를 명시적으로 결정해야 한다

데이터를 다루는 방식에는 두 가지 철학이 있다.

**읽을 때마다 계산 (Compute on Read):**
```typescript
// CartItem에 rawPrice와 options만 저장
// 렌더링할 때마다 totalPrice를 계산
const totalPrice = calculateItemTotalPrice(item.price, item.options);
```

**저장할 때 계산 (Compute on Write):**
```typescript
// 장바구니에 담을 때 이미 계산된 값을 저장
const cartItem = {
  unitPrice: basePrice + optionsTotalPrice,  // 미리 계산
  displayLabels: options.map(o => o.label),  // 미리 계산
};
// 렌더링할 때는 그냥 item.unitPrice 사용
```

장바구니처럼 **자주 읽히고 드물게 쓰이는 데이터**는 저장 시점에 계산하는 게 합리적이다.
렌더링마다 재계산이 없어 코드가 단순해지고, 헬퍼 함수 의존성도 사라진다.

**기억할 것:**
`calculate*` 헬퍼 함수가 필요해지는 순간, "이 계산을 저장 시점으로 옮길 수 없을까?"를 먼저 생각해보자.
데이터 모델이 잘 설계되면 로직이 단순해진다.

## 5. 규모에 맞는 도구를 고르는 것도 실력이다

기술은 더 강력할수록 더 많은 보일러플레이트를 요구한다.
React Query는 캐싱, 재시도, 무효화, 로딩 상태 관리를 제공하는 강력한 도구다.
하지만 그 가치는 **동일한 데이터를 여러 컴포넌트에서 공유하거나 캐시가 필요할 때** 드러난다.

단일 페이지에서 한 번 불러오고 끝나는 데이터라면:
```typescript
// 이것으로 충분하다
const [data, setData] = useState(null);
useEffect(() => { http.get('/items').then(setData) }, []);
```

React Query를 추가하면 `QueryClient`, `QueryClientProvider`, query key 관리, `useQuery` 래핑...
이 모든 설정이 생기는데, 과제에서 얻는 이점은 거의 없다.

**기억할 것:**
도구를 고를 때 "이 도구가 제공하는 기능 중 내가 실제로 쓰는 게 몇 %인가?"를 따져보자.
20%만 쓴다면 더 단순한 도구가 정답이다.

## 정리

| 함정 | 바른 방향 |
|------|-----------|
| 모든 기술을 꺼내 쓴다 | 맥락에 맞는 도구만 고른다 |
| 정상 흐름만 구현한다 | 경계 조건까지 고민한다 |
| 추상화를 많이 만든다 | 의도가 바로 보이는 코드를 쓴다 |
| 읽을 때마다 계산한다 | 저장 시점에 계산을 끝낸다 |
| 파일 수 = 설계 수준 | 파악 속도 = 설계 수준 |
